# Evaluation prompt set

`prompts.jsonl` is the frozen `baseline-v1` prompt set. Do not edit it in
place after generating baseline audio; create a new version instead.

## Cohorts

- `prod_original`: the five production prompts, preserved verbatim.
- `prod_adjusted`: paired, music-focused versions of the production prompts.
- `dataset_eval`: twenty manually curated prompts assigned to the deterministic
  held-out split used by `prepare.py`.

Scores should be reported per cohort as well as across the complete set.

## Provenance

Production prompts:

- Repository: `https://github.com/kennethnym/infinifi`
- Revision: `eba64617d99eaff3ee31d5fd0e192128e320e14f`
- File: `prompts.py`

Dataset prompts:

- Dataset: `vikhyatk/lofi`
- Revision: `966a2d3065aac26c0385b4ef2d50983c0429a305`
- Source shard:
  `data/dynamic-prompts-00039ef5-5aed-4d51-8591-4c8f9fb7f6fd.parquet`
- A source ID is in the eval split when the first eight bytes of its SHA-256
  digest, interpreted as an unsigned big-endian integer and divided by
  `2**64`, are greater than or equal to `0.8`.

The dataset audio was generated by MusicGen Large. Any FAD measurement using
that audio as the reference measures similarity to the synthetic dataset
domain, not general realism against recorded music.

## Integrity

The expected SHA-256 digest of `prompts.jsonl` is recorded in
`prompts.sha256`.

## Generate the plain-model baseline

Validate the frozen prompt set and inspect the run configuration:

```bash
python eval/generate.py \
  --model facebook/musicgen-small \
  --run-name baseline_musicgen_small \
  --dry-run
```

Generate the complete baseline:

```bash
python eval/generate.py \
  --model facebook/musicgen-small \
  --run-name baseline_musicgen_small
```

The run generates all 30 prompts with the default seeds 42, 43, 44, and 45 for
120 WAV clips. Generation uses a 30-second duration, sampling enabled, CFG 3.0,
temperature 1.0, top-k 250, top-p disabled, and WAV loudness normalization.
Outputs are written to `runs/baseline_musicgen_small/`.

## Generate a fine-tuned checkpoint run

Keep Dora checkpoint export separate from audio generation. After training,
set the signature printed by Dora, export the LM with the
`facebook/encodec_32khz` compression model, and pass that package directory to
the same generator:

```bash
DORA_SIGNATURE=aec31258
python export_checkpoint.py \
  --signature "$DORA_SIGNATURE" \
  --output-dir /checkpoints/infinifi
python eval/generate.py \
  --model /checkpoints/infinifi \
  --run-name finetuned_infinifi
```

Use the actual signature from the training run. The exporter refuses to
overwrite a non-empty directory unless it is already a complete export for the
same signature with matching file hashes.

The unified generator loads both pretrained IDs and local packages with
`MusicGen.get_pretrained(...)`, so baseline and fine-tuned runs use identical
prompts, seeds, generation parameters, and audio normalization. Config and
manifest records identify the supplied model source. Local packages also get a
deterministic digest over all regular package files; pretrained IDs are tied to
the pinned AudioCraft commit.

The generator writes each manifest record immediately after its WAV file.
Running the same command again safely resumes an interrupted run and refuses
to continue if the locked configuration or runtime environment has changed.

For a one-clip smoke test, use a distinct run name:

```bash
python eval/generate.py \
  --model facebook/musicgen-small \
  --run-name baseline_smoke \
  --limit 1 \
  --seeds 42
```

`--run-name` is a validated identifier and always maps directly to
`runs/<run-name>/`; it cannot select an arbitrary output path.

## Score a completed run

`score.py` reads the locked run configuration and `manifest.jsonl`; it does not
regenerate audio or modify Dora's `history.json`. Validate a complete run and
the held-out references first:

```bash
python eval/score.py \
  --run-name baseline_musicgen_small \
  --dry-run
```

Then compute the default metric suite:

```bash
python eval/score.py \
  --run-name baseline_musicgen_small
```

The scorer computes:

- CLAP text consistency for every generated clip and reports results overall
  and per cohort.
- PaSST KLD for `dataset_eval`, pairing every generated clip with the held-out
  source audio identified by `source_id`.
- VGGish FAD for the generated and reference `dataset_eval` corpora.

By default, references are loaded from
`audiocraft/dataset/lofi/eval`. Pass `--reference-dir` if prepared data is
stored elsewhere. All twenty frozen `dataset_eval` source IDs must be present;
increase `BATCH_COUNT` and rerun `prepare.py` if the scorer reports missing
references.

CLAP uses LAION's `music_audioset_epoch_15_esc_90.14.pt` checkpoint with the
HTSAT-base architecture. The first scoring run downloads the approximately
2.35 GB checkpoint into the Hugging Face cache. To use a pre-downloaded copy:

```bash
python eval/score.py \
  --run-name baseline_musicgen_small \
  --clap-checkpoint /path/to/music_audioset_epoch_15_esc_90.14.pt
```

The checkpoint is accepted only when its SHA-256 digest matches the frozen
checkpoint. The repository's Docker image includes the CLAP, PaSST, and
VGGish packages needed by the scorer. PaSST and VGGish also download their
pretrained weights into the Torch cache on first use.

For constrained runs, select individual metrics:

```bash
python eval/score.py \
  --run-name baseline_musicgen_small \
  --metrics clap
```

Scoring writes the following files beside the generated run:

```text
runs/<run-name>/
  score_config.json
  clip_metrics.jsonl
  metrics.json
```

`clip_metrics.jsonl` contains CLAP for every clip and KLD values for clips with
references. `metrics.json` contains aggregate scores and per-cohort CLAP
summaries. FAD is corpus-level and therefore appears only in `metrics.json`.
`score_config.json` locks input manifests, generated and reference audio
digests, the CLAP checkpoint, metric package/runtime settings, and scorer code.

Existing complete results are reused only when their locked configuration
matches. Use `--overwrite` to deliberately recompute them.

The FAD implementation uses PyTorch VGGish pre-PCA embeddings without the
final activation. It closely follows the common PyTorch FAD configuration but
is not byte-for-byte identical to AudioCraft's external TensorFlow wrapper.
With only twenty unique reference tracks, FAD should be treated as a noisy
comparison between runs. Also, these references were generated by MusicGen
Large, so FAD measures similarity to that synthetic dataset domain rather than
general recorded-music realism.
